# Go 与 Shell 代码英文化计划

## 目标

对仓库内所有 Go（`*.go`）与 Shell（`*.sh`）文件进行“英文化”处理：除了注释以外，代码中不允许出现中文字符（汉字）。为保证可验证性与一致性，本次执行将采用更严格的标准：**在 `*.go` 与 `*.sh` 文件中完全不出现任何中文字符（包括注释）**。

这样可以避免“注释/非注释”的解析边界问题（例如结构体 tag、Swagger 注解、raw string、shell heredoc 等），并能用简单的文本扫描作为验收手段。

## 范围

- 需要处理：
  - `**/*.go`
  - `**/*.sh`
- 不处理（不在本任务强制范围内）：
  - `docs/**`、`README` 等纯文档
  - `database/**` 里的迁移 SQL / HCL（除非是 `.go` / `.sh`）
  - 其他非 Go / 非 Shell 的资源文件（即使包含中文也允许存在）

## 主要改动类型（将统一改为英文）

1. **字符串字面量**
   - API 响应文案、错误信息、日志事件字段、CLI 帮助文案、测试断言信息等。
2. **结构体 tag / Swagger 注解**
   - 如 `swagger:"required,用户姓名"`、`@Description ...` 等，这些属于代码字符串/注解，不属于 Go 注释语法意义上的注释，按严格标准一律英文化。
3. **i18n/校验器内置的中文翻译/消息**
   - 当前在 Go 文件中直接内置中文翻译或中文提示，会违反“代码不得出现中文”的要求。
   - 处理策略优先级：
     - A. 将中文资源外置到非 Go 文件（例如 `resources/i18n/zh.json`），Go 侧只保留英文 key 与加载逻辑（推荐，保留 i18n 能力）。
     - B. 若项目并不需要中文翻译，则只保留英文（最简单，但会改变功能）。

## 执行步骤

1. **建立基线扫描**
   - 扫描所有 `*.go` 与 `*.sh` 中的中文字符出现位置，按文件/模块归类（controllers、middleware、services、cli、tests、i18n、scheduler、helpers）。
2. **统一英文文案风格**
   - 规则：
     - 错误消息：简洁、可读、面向客户端（例如 `invalid request body`）。
     - 日志 message：保持既有 key（如已存在），但文案改英文；不要输出敏感信息。
     - CLI 文案：`Short`/`Long` 改为英文，保持语义一致。
3. **逐模块替换中文内容**
   - `internal/transport/http/**`：API 返回、Swagger 注解、请求验证错误等。
   - `internal/services/**`：业务错误、日志。
   - `internal/cli/**`：cobra 命令说明、输出文案。
   - `internal/i18n/**`：按“外置中文资源”策略重构，确保 Go 文件不包含中文。
   - `tests/**`：测试用例消息改英文（避免中文字符串触发扫描失败）。
   - `scripts/*.sh`：echo/usage 文案改英文。
4. **格式化与静态检查**
   - `gofmt` 处理所有改动 Go 文件。
   - `go test ./...` 与 `go vet ./...` 确保通过。
5. **最终强制扫描验收**
   - 对 `*.go` 与 `*.sh` 进行全仓扫描，确保 **0 个中文字符匹配**。

## 验收标准

- `go test ./...` 通过
- `go vet ./...` 通过
- 在仓库根目录执行等价扫描时：
  - `**/*.go` 无任何中文字符
  - `**/*.sh` 无任何中文字符

## 风险与兼容性说明

- API 与 CLI 的对外文案会从中文变为英文，可能影响依赖文案的下游（例如前端提示、自动化测试、文档截图）。
- i18n 若选择“只保留英文”，会移除中文翻译能力；若选择“外置中文资源”，会新增资源文件与加载逻辑，但能保留功能。

## 回滚策略

- 本次改动主要为文案与资源组织调整，不涉及数据库结构变更。
- 若需要回滚，可按模块逐步回退对应文件或恢复原文案。
