# 结构化高可用重构计划

## 背景与目标

当前项目已经具备较完整的分层（controllers/middleware/services/database/config/providers），但在“高可用（HA）”与“结构化（可维护/可测试/可演进）”方面存在一些典型的系统性风险：启动/退出流程、健康检查语义、错误处理链路不唯一、日志不统一、全局单例（config.GlobalConfig / database.DB）与 DI 并存、队列 worker 生命周期未纳入统一运维模型等。

本计划目标：

* 让服务具备生产级 HA 基线：优雅退出、readiness/liveness、可观测性、外部依赖降级策略。

* 让工程更结构化：单一错误出口、统一日志字段、消除关键全局状态、明确边界（HTTP/业务/基础设施）。

* 让后续重构可控：分阶段（P0/P1/P2）、每阶段可验证、有回滚策略。

非目标（本轮不做/或仅做最小改动）：

* 暂不推进 Postgres 迁移/验证（按你的要求先跳过）。

## 现状快速体检（关键发现）

1. middleware 里存在未初始化的全局 Logger 使用风险：

* `app/http/middleware/setup.go` 内 `var Logger *zap.Logger` 未初始化，但在错误处理中调用。

1. 启动逻辑包含“kill 端口进程”的行为：

* 该行为在生产场景不安全，不适合滚动升级与容器编排。

1. 可观测性链路不统一：

* Fiber 的 access log 与 zap 结构化日志不统一，request\_id 传播与字段对齐不完整。

1. 全局状态与 DI 并存：

* `config.GlobalConfig`、`database.DB` 等全局变量与 dig 注入同时存在，增加测试难度与多实例风险。

1. 队列（asynq）与主进程生命周期未对齐：

* worker 未提供明确的 CLI 运行模式（更适合独立进程/部署单元）。

### 建议新增/整理的 rules（让项目更“结构化”）

建议把项目规则写成可执行的“工程契约”（不是口号），并纳入仓库（例如 `.cursor/rules` / `.trae/rules` 或你团队常用目录）。重点规则建议：

1. 单一错误出口

* HTTP 层统一由一个错误处理中间件/handler 输出响应结构，禁止散落的 `ctx.Status(...).JSON(err)` 风格。

1. 禁止关键全局单例

* 禁止业务层直接使用 `config.GlobalConfig` / `database.DB`；统一从 DI 注入或显式参数传递。

1. 统一日志字段

* request\_id、user\_id、trace/span（即便暂不引入 tracing，也要预留字段），并保证每条错误日志包含关键上下文。

1. 依赖可用性策略

* DB/Redis/外部服务的超时、重试、降级策略必须显式；测试默认不依赖外部服务。

1. 生成代码管理规则

* `internal/db/gen/**` 作为生成物，必须有固定生成命令与版本锁定策略；禁止手改生成文件。

## 分阶段重构路线图（P0 / P1 / P2）

### P0：稳定性基线（必须做，改动可控）

P0-1 收敛错误处理链路为唯一真源

* 目标：统一响应结构与日志行为，避免多套 handler 冲突

* 动作：

  * 选择 `app/http/middleware/error_handler.go` 作为唯一出口

  * 让 Fiber `Config.ErrorHandler` 仅做兜底并转交

  * 删除/合并 `SetupErrorHandling` 中重复 404/全局错误逻辑

P0-2 修复 middleware Logger 风险并统一日志入口

* 目标：消除潜在空指针；所有错误日志走同一 zap logger

* 动作：

  * 删除 middleware 包内的全局 `Logger *zap.Logger`

  * 统一使用 `helpers.Logger` 或从 DI 注入 logger

P0-3 优雅退出（Graceful Shutdown）

* 目标：SIGTERM/SIGINT 可平滑退出；释放 DB/Redis/queue/client 资源

* 动作：

  * main/bootstrap 引入 signal 监听

  * Fiber 优雅 shutdown + 等待 in-flight

  * 统一 Close 顺序：HTTP → Queue → Cache/Redis → DB → Logger.Sync

P0-4 健康检查语义化（liveness/readiness）

* 目标：部署平台可据此做滚动升级与自动恢复

* 动作：

  * `/health`：只反映进程活着

  * `/ready`：检查关键依赖（sqlite 文件可写、DB ping、redis 可选）

验证标准：

* `go test ./...` 通过

* 本地启动后：`/health` 恒 200；`/ready` 在依赖不可用时返回非 200 且结构化错误

### P1：结构化工程化（可维护/可测试）

P1-1 去全局状态，收敛到 DI

* 目标：提升可测性，支持多实例/多环境

* 动作：

  * 逐步去掉 `database.DB` 直接引用；改由注入 `*sql.DB` 或 `database.Connection`

  * 逐步减少 `config.GlobalConfig` 在业务层出现

P1-2 统一 HTTP access log 与 request\_id 传播

* 目标：日志可检索、可关联；定位问题成本下降

* 动作：

  * 用 zap 字段化输出 access log（或在现有 Fiber logger 中补齐 request\_id）

P1-3 队列 worker 运维模型

* 目标：worker 作为独立命令/进程可横向扩展

* 动作：

  * 新增 `queue:work` cobra 命令启动 asynq server

  * 明确队列连接与重试策略

### P2：HA 与发布体系完善（可选但推荐）

P2-1 CI 基线（最小可用）

* `go test ./...`

* `go vet ./...`

* 可选：golangci-lint

* 可选：生成物一致性校验（bobgen 输出是否过期）

P2-2 观测能力升级（按需要）

* 指标：latency、error rate、依赖可用性（ready）

* tracing：视你们链路追踪平台决定（OpenTelemetry 方案需另开一轮设计）

## SQLite / Atlas / bobgen 的结构化落地建议

1. Atlas 作为 SQLite 迁移唯一来源

* 迁移目录固定：`database/migrations/atlas/sqlite`

* schema-as-code 固定：`database/schema.lt.hcl`

1. bobgen 生成策略

* 生成命令固定在 Makefile

* 生成输出目录固定在 `internal/db/gen/sqlite`

* 在 CI 中增加 “生成一致性” 检查（P2）

## 执行顺序（建议）

1. P0 全部完成并稳定（优先保证线上风险最低）
2. P1 逐步替换全局依赖（每次只动一个服务/一条链路）
3. P2 引入 CI 与观测增强
